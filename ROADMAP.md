# Дорожная карта (Roadmap)

Цель: рабочий, безопасный и идемпотентный платёжный бэкенд с тестами и README.
Принцип: каждый этап завершается зелёными тестами и работающим состоянием.

---

> ⚠️ **Вне scope (из ТЗ «что НЕ нужно делать»):** Docker/CI/CD/деплой;
> реальная интеграция с платёжкой (только заглушки); аутентификация/регистрация.

## Фаза 0 — Подготовка (≈0.5 ч)

- Инициализация репозитория (Node + TS + ESLint/Prettier).
- `.env.example`; MongoDB и Redis поднимаются локально разработчиком (без Docker).
- Каркас Express (`app.ts`/`server.ts`), health-check `GET /health`.
- **DoD:** приложение поднимается, подключается к Mongo и Redis, `/health` отвечает.

## Фаза 1 — Доменные модели и деньги (≈1 ч)

- Модель `Merchant` (`feePercent`), `Invoice`, `WebhookEvent` (nonce/eventId).
- `lib/money.ts`: расчёт `fee` и `amountToReceive` в **минорных единицах**, явное округление.
- **DoD:** unit-тесты на расчёт комиссии (включая округление и граничные суммы).

## Фаза 2 — Создание инвойса (≈1 ч)

- `POST /invoice`: валидация DTO (`amount`, `currency`, `merchantId`) →
  расчёт `fee`, `amountToReceive` → сохранение со статусом `pending` → ответ `invoiceId` + суммы.
- `GET /invoice/:id`: возврат статуса (404 если нет).
- **DoD:** integration-тесты на создание и чтение, проверка сумм.

## Фаза 3 — Безопасность вебхука (≈1.5 ч)

- Сохранение **raw body** для подписи (middleware до JSON-парсинга).
- `lib/hmac.ts`: `HMAC-SHA256(secret, rawBody)`, сравнение через `timingSafeEqual`.
- Проверка `X-Timestamp` (в пределах окна `WEBHOOK_TIMESTAMP_TOLERANCE_SEC`).
- Проверка `X-Nonce` на уникальность (Redis `SET NX` + TTL, дублируем в Mongo для надёжности).
- Невалидная подпись/просроченный timestamp/повторный nonce → `4xx` без побочных эффектов.
- **DoD:** тесты на: валидную подпись (200), битую подпись (401), старый timestamp (401), повтор nonce (409/200-noop).

## Фаза 4 — Идемпотентное зачисление (ядро) (≈2 ч)

- `POST /webhook` payload `{ invoiceId, status }`.
- Переход статуса `pending → paid|failed` **атомарно** и **один раз**:
  - вариант: атомарный `findOneAndUpdate({_id, status: 'pending'}, {status})` —
    зачисление выполняется только если документ реально перешёл из `pending`;
  - distributed lock в Redis на `invoiceId` для сериализации конкурентных вебхуков;
  - запись факта обработки события (`webhookEvent`) — повтор не зачисляет снова.
- При `paid` — побочный эффект «зачисление» (лог/поле `settledAt`) ровно один раз.
- **DoD:** тесты на: один webhook → одно зачисление; N одновременных дублей → ровно одно зачисление; `failed` не зачисляет.

## Фаза 5 — Надёжность, наблюдаемость, оформление (≈1 ч)

- Глобальный error-handler, единый формат ошибок, корректные HTTP-коды.
- Структурное логирование (pino), валидация конфига при старте.
- README: запуск, переменные окружения, запуск тестов, **принятые допущения** и
  **что не доделано + как бы доделал**.
- **DoD:** `npm test` зелёный, приложение запускается локально по README, README полон.

## Карта рисков → решений

| Риск                                        | Решение                                                          |
| ------------------------------------------- | ---------------------------------------------------------------- |
| Потеря точности денег (`float`)             | Целые минорные единицы / Decimal128, явное округление            |
| Двойное зачисление при дубле/гонке вебхуков | Атомарный условный апдейт по статусу + Redis lock + лог события  |
| Replay-атака                                | HMAC от raw body + окно timestamp + одноразовый nonce (TTL)      |
| Подмена тела при пересборке JSON            | Подпись считается от **сырого** тела, не от пересериализованного |
| Тайминг-атака на сравнение подписи          | `crypto.timingSafeEqual`                                         |
| Падение Redis                               | Fallback-дедуп через unique-индекс в Mongo (`nonce`/`eventId`)   |

## Последовательность зависимостей

```
Фаза 0 → Фаза 1 → Фаза 2
                    ↓
                 Фаза 3 → Фаза 4 → Фаза 5
```

Ориентировочно ~7–8 часов. MVP = Фазы 0–4; Фаза 5 — полировка и оформление.
